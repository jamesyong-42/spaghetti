Spaghetti Parser Engine — Proposed 4-Layer Architecture v2

Refactor proposal post-RFC 005 · 2026-04-29 · iterate via this page

Layers

L1 Public API
What consumers call. Pure facade. Knows L2 only.
L2 Domain Services
DataStore = state of the world. HotUpdate = deltas to that state. Two siblings.
L3 Composing Services
Lifecycle + heavy lifting. Cache, LiveComposer, Pipeline, QueryEngine, Bus.
L4 Atomic Services
Pure / stateless-ish. File, sqlite, parsers, watchers, native bridge.

Key boundary fixes

  • DataStore owns reads + caches + subscriber bus. Does NOT own writes — writes notify it via apply(change).
  • HotUpdate owns the entire live pipeline. Reaches DataStore only through apply().
  • CacheService owns ingest engine selection (TS/RS), cold/warm start, idle compaction, worker pool.
  • IngestPipeline (L3) is owned by CacheService, borrowed by LiveComposer. Single writer, single statement cache.
  • NativeAddon (L4) hides the ingest() vs live_ingest_batch() split from upper layers.

Key flows

Bootstrap (cold): API → DataStore.init() → Cache.bootstrap() → [WorkerPool ‖ Native] → IngestPipeline → SQLite → DataStore.refresh()

Live delta (warm): Watcher → LiveComposer (Queue → IncrementalParser) → IngestPipeline → DataStore.apply(change) → SubscriberBus → listeners

Read: API → DataStore → QueryEngine → SQLite

Notation

*-- composition · o-- aggregation · ..|> realization · ..> dependency

Open questions / iteration points
  • DataStoreService.apply vs notify: should writes flow through DataStore (it calls IngestPipeline) or alongside (CacheService/LiveComposer write, then notify)? Diagram shows alongside — DataStore stays read+cache+bus, never holds the writer. Tradeoff: simpler dependency graph but two write entry points instead of one.
  • ProjectParseSink: proposed to fold it into IngestPipeline.applyParseEvent(event) rather than keep it as a separate interface. Alternative: keep the sink so a TS-only test can drive IngestPipeline without the full pipeline.
  • WorkerPool ownership: shown owned by CacheService. Alternative: extract a ColdStartRunner (L3) that owns pool + parser invocation, so CacheService becomes pure orchestration.
  • HotUpdateService thinness: currently it's almost a pass-through to LiveComposer. Could collapse them into one L2 service. Kept separate to keep "policy" (L2 — when/whether) distinct from "mechanism" (L3 — how).
  • NativeAddon facade: should L3 IngestPipeline see two methods (ingest, liveIngestBatch) or one (write(events))? Latter is cleaner but means hiding cold-vs-warm semantic at L4.
  • Cold-start engine='rs': currently bypasses IngestPipeline entirely (Native does parse+write). Proposed: keep that bypass for performance, but route progress events through CacheService so DataStore is notified consistently.
What's intentionally NOT in this diagram
  • Type modules (packages/sdk/src/types/, crates/spaghetti-napi/src/types/).
  • React adapter (packages/sdk/src/react/) — pure consumer.
  • Channel + hooks plugin packages — separate MCP concerns.
  • App-level wiring (apps/, CLI, Electron) — calls createSpaghettiService.
  • Legacy segment-store / segment-types — deprecated; would be deleted in this refactor.
